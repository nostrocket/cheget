<objective>
Wire `keygen` to persist the full DKG share set through the Phase 2 participant store.
Today `cli/keygen.rs` discards the secret shares (`let (_shares, group) = ...`) and writes only
the public envelope; this plan adds the secret-persistence path so `keygen` becomes the first
command that creates an encrypted store — 100 separate per-seat roots, each with its `KeyPackage`
age/scrypt-encrypted plus the plaintext group package, under one prompt-once passphrase (D-02/D-03/D-04).

Purpose: Deliver the write half of KEY-06's persist/reload-through-real-commands criterion and
unblock the outstanding Phase 2 store-creation UAT (Test 1), which is blocked only because no
command yet drives `InteractivePassphrase::for_new_store`.
Output: A prompt-once passphrase reuse seam (`ResolvedPassphrase`), the re-exported
`InteractivePassphrase`, a rewired `keygen::run` that persists per-seat roots, and a small-n
writer-correctness test.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| operator terminal -> process | The store passphrase is entered interactively; it must never be echoed, logged, or written to disk. |
| process memory -> disk (store) | Secret KeyPackage bytes cross to disk; they must cross ONLY inside age/scrypt ciphertext. The group public package crosses in plaintext by design. |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-03-01 | Information Disclosure | keygen persist loop writing 100 shares | high | mitigate | Shares are written only via `ParticipantStore::put_share`, which serializes into `Zeroizing`, age/scrypt-encrypts, and atomic-writes at 0600 (participant.rs). Task 3 asserts a wrong passphrase cannot decrypt (byte-equal only under the same passphrase). No plaintext share path is added. |
| T-03-02 | Information Disclosure | ResolvedPassphrase / prompt-once reuse | high | mitigate | The passphrase is acquired once via `acquire_store_passphrase(true)` and reused via `ResolvedPassphrase` clones; it is never written to disk or logs and never sourced from env/flag (D-01/D-03). Completion output prints only t/n/base, never the passphrase. |
| T-03-03 | Tampering | store path from key_id/seat | medium | mitigate | `KeyId`/`ShareTag` reuse the validated crypto newtypes; `KeyId::active()` and `Epoch::GENESIS` are known-safe path components (types.rs path-traversal guard T-02-12). Per-seat root name is `format!("seat-{:04}", i)` from a bounded integer. |
| T-03-04 | Repudiation/Integrity | half-written share vs manifest | low | accept | `put_share` already enforces D-07 write ordering (share ciphertext durable before manifest); this plan reuses it unchanged. |
| T-03-SC | Tampering | dependencies | n/a | accept | No new crates are added in this phase — supply-chain surface is unchanged; existing `Cargo.lock` pins hold. |
</threat_model>
