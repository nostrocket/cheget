Define the `ChainBackend` trait (sync, per RESEARCH Open Q2) and two implementations — Bitcoin
Core JSON-RPC (fronts the confirmed-key-spend path, D-07) and Esplora (built to the same trait,
conformance-tested only) — plus the BIP341 key-spend sighash helper and the auto-spawned regtest
fixture (corepc-node, D-05). This is STOR-04 and the chain seam that 01-04's signing session and
Phase 5's sweep/watch reuse.

n=1000 confirmed key-spend can run hermetically in CI. A wrong sighash type silently changes the
challenge — the sighash helper hard-wires SIGHASH_DEFAULT over all prevouts.
Output: `chain/` (trait + two backends + sighash helper), regtest fixture, conformance tests.
</objective>

| Boundary | Description |
|----------|-------------|
| PSBT / prevout data → sighash | Wrong sighash type or missing prevout silently changes the signed challenge |
| downloaded Core binary → test host | Supply-chain of the auto-spawned regtest node |
| chain RPC endpoint → local state | Untrusted UTXO/fee data from a node/Esplora endpoint |

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |

| T-01-sighash | Tampering | `chain/sighash.rs` | high | mitigate | `TapSighashType::Default` + `Prevouts::All` hard-wired; single helper reused for client-side recompute (SIGN-07) |
| T-01-coredl | Tampering | corepc-node Core binary | medium | mitigate | Auto-download hash-verifies a pinned Core version (D-05); dev-dependency only |
